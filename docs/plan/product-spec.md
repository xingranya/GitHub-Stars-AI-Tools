# 产品规格：GitHub-Stars-AI-Tools

## 产品定位

一个面向重度 GitHub 用户的独立 Star 管理工作台，把“曾经 Star 过但忘记用途的项目”转化为可搜索、可理解、可追溯的个人 AI 知识库。

## 用户画像

- 经常 Star 开源项目，但缺少长期整理习惯的开发者。
- 需要快速找回工具、库、模板、AI 项目的工程师。
- 英文 README 阅读成本高，希望先看中文摘要再决定是否深入的人。
- Star 数量达到数百到数千，GitHub 原生列表已经难以管理的用户。

## 核心问题

1. Star 项目太多，GitHub 原生列表难以搜索和分类。
2. 用户经常忘记当初为什么 Star 某个项目。
3. 英文 README 降低回访效率。
4. 简单关键词无法表达真实需求，例如“找一个能把网页转 Markdown 的工具”。
5. 标签和笔记需要长期保存，不能被同步覆盖。

## 核心用户流程

### 首次使用

1. 打开应用。
2. 连接 GitHub 账号。
3. 选择同步范围：全部 Stars。
4. 应用同步仓库元数据。
5. 应用后台抓取 README 并生成中文摘要。
6. 用户进入 Star 工作台。

### 日常整理

1. 用户搜索或筛选项目。
2. 批量添加标签。
3. 给关键项目添加笔记。
4. 对 AI 推荐标签进行确认或忽略。
5. 应用自动保存整理结果。

### 自然语言找项目

1. 用户输入：“找一个适合做本地 RAG 的向量数据库”。
2. 系统召回相关项目。
3. 系统展示：项目名、中文用途说明、匹配原因、README 引用片段、标签、语言。
4. 用户打开项目详情查看中文摘要和原 README。

## 信息架构

- 首页/工作台
  - Star 总览
  - 搜索框
  - 语言筛选
  - 标签筛选
  - 未整理项目入口
  - 最近同步状态

- AI 检索页
  - 自然语言输入框
  - 推荐查询示例
  - 匹配结果
  - 匹配原因
  - 引用片段

- 项目详情页
  - 基础信息
  - 中文摘要
  - README 原文
  - 标签
  - 笔记
  - GitHub 链接

- 标签管理页
  - 标签列表
  - 合并标签
  - 重命名标签
  - 删除标签
  - 批量应用

- 同步中心
  - Star 同步状态
  - README 抓取状态
  - AI 处理队列
  - 失败任务
  - 重试操作

- 设置页
  - GitHub 账号
  - AI 服务
  - OpenAI、OpenAI 兼容接口和 Anthropic 配置
  - 数据导入导出
  - 成本与用量

## 当前上线范围

- 桌面应用基础壳。
- GitHub Personal Access Token 输入、校验与系统凭据管理器安全存储。
- 全量同步与增量同步 Star 元数据。
- Star 列表、关键词搜索、语言筛选、标签筛选。
- 标签、笔记、批量打标。
- README 抓取与缓存。
- OpenAI、OpenAI 兼容接口与 Anthropic 配置，支持用户自定义请求地址、API Key 和模型 ID。
- 中文摘要、关键词与推荐标签生成。
- 自然语言上下文检索。
- AI 标签网络生成。
- 基于选中 Stars 的 GitHub 相似项目发现。
- Gist 注解导入导出。
- 任务进度、失败重试、AI 用量统计和发布包真实链路自检。
- 项目详情页展示中文摘要和 README 原文。
- 本地 SQLite 持久化。

## 后续增强范围

- README 中文全文翻译。
- zvec 本地向量索引与混合语义检索。
- 自动更新。
- 更细粒度的成本预算和批量任务暂停/恢复。
- 可选云同步。

## 长期方向

- 混合检索：FTS + 向量 + rerank。
- 多 AI 服务配置：OpenAI、OpenAI 兼容接口与 Anthropic。
- 后续新增 AI 服务时继续沿用统一请求配置，不影响业务层。
- 多账号。
- 团队协作功能。

## 非目标

- 不做浏览器插件主线。
- 不做 GitHub 页面嵌入式体验。
- 不做团队协作文档库。
- 不默认上传用户笔记给 AI。

## 成功指标

- 1000 个 Star 项目可在本地稳定同步和浏览。
- 关键词搜索响应小于 300ms。
- 项目详情页能清晰解释仓库用途。
- 自然语言检索 Top 10 中至少 5 个结果与用户意图相关。
- README 摘要重复生成率接近 0，内容未变更不重复扣费。
