# AI 检索质量审查

## 审查基线（修复前）

审查时桌面端并未使用向量检索。`repo_embeddings` 表、TypeScript `VectorIndexPort` 和内存基准实现没有接入 Tauri Rust 搜索命令，因此自然语言搜索仍是“AI 扩写关键词 + 全库字符串包含评分”。这会让问候语、泛化词和上一轮上下文产生大量弱相关结果。

## 已确认问题

1. `search_repositories_worker` 在判断用户是否真的要搜索之前调用 AI 扩写，`你好` 等对话内容可能被扩写为宽泛技术词。
2. `score_search_row` 对上一轮仓库无条件加分，即使本轮查询没有任何直接命中，也会继续返回旧结果。
3. 中文查询会生成全部二元字符片段，字段只要包含任意片段就能累计高权重，缺少最低相关度门槛。
4. 后端允许单次返回 1 至 100 条，并把全量弱命中数作为 `totalCount` 提供给分页；前端每页 6 条，因此会出现数百个结果。
5. 当前排序主要依赖字符串包含和 Stars 数，不能召回同义表达，也不能可靠排除语义不相关项目。
6. SQLite 已保存 embedding schema，但 Rust 主链路没有 embedding 请求、索引写入、zvec adapter、索引恢复或状态检查。
7. AI 搜索最终回复只是“找到 N 个仓库”，没有基于检索结果生成受证据约束的回答。

## S.U.P.E.R 评估

| 原则 | 当前评价 | 主要问题 |
| --- | --- | --- |
| S 单一职责 | 部分符合 | `storage.rs` 同时承担查询、评分和结果解释 |
| U 单向数据流 | 部分符合 | 共享 TypeScript 向量能力与实际 Rust 主链路形成两套未连接实现 |
| P 接口优先 | 不符合 | 桌面端缺少 embedding 配置、向量记录和检索状态契约 |
| E 环境无关 | 部分符合 | 向量索引路径和原生库打包策略尚未实现 |
| R 可替换部件 | 部分符合 | roadmap 有 Port 设计，但 zvec 尚无 Rust adapter 与 SQLite 回退 |

## 修复目标

- 对问候、感谢和纯对话请求返回 0 个仓库，并提示用户描述项目需求。
- 所有搜索最多返回 10 条，默认返回 8 条；弱相关项在后端直接淘汰。
- 配置 embedding 服务后使用 zvec cosine TopK 召回，并与严格关键词分数融合。
- zvec 仅作为可重建索引；SQLite 继续保存 embedding 元数据和向量事实。
- zvec 不可用、索引为空或 embedding 请求失败时，自动回退到严格关键词检索。
- AI 可用时基于最终候选生成简短回答；无 AI 时提供可核验的本地摘要。

## 修复结果

以上问题已在 Phase 10 完成修复：问候语执行确定性门控，英文技术词按完整词边界匹配，短缩写要求关键词确认，多词查询要求至少命中多个词；严格关键词和 zvec 向量候选融合后默认返回 8 条、硬上限 10 条。SQLite 保存向量事实，zvec 可从 SQLite 恢复；最终回答只使用 Top 结果的仓库名、描述、Topics 和摘要，个人笔记不会发送给远程 AI。
